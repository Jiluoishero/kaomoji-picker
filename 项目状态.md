# 项目状态

## 当前定位

这是一个 Windows 桌面颜文字/符号快速输入工具。程序常驻系统托盘，通过全局快捷键唤起一个 PySide6 原生浮动面板，用户点击符号后，程序把符号粘贴回唤起面板前的目标输入窗口。

当前项目已经从早期 `pywebview + WebView2` 方案迁移到 PySide6。迁移后的目标不是跨平台通用窗口，而是一个更接近软键盘行为的 Windows 小工具：

- 面板以无边框、圆角、置顶浮窗形式显示。
- 默认不抢占目标输入框焦点。
- 支持全局快捷键唤起。
- 支持单次模式和固定模式。
- 支持多屏定位、标题栏拖动、窗口大小保存。
- 支持分组、添加、删除、重命名、拖拽排序、跨组移动。
- 点击符号后使用剪贴板和模拟 `Ctrl+V` 粘贴回目标窗口。

## 当前技术栈

- 语言：Python 3.10+
- GUI：PySide6 / Qt 原生窗口
- 运行平台：Windows
- 全局快捷键：Win32 `RegisterHotKey`
- 数字键切组：Win32 low-level keyboard hook
- 窗口焦点策略：Win32 window extended style / `WS_EX_NOACTIVATE`
- 剪贴板：`pyperclip` + `win32clipboard`
- 粘贴模拟：Win32 `keybd_event` 模拟 `Ctrl+V`
- 系统托盘：`QSystemTrayIcon`
- 数据存储：`data.json`
- 用户配置：运行时生成的 `config.json`
- 开机自启：当前用户注册表 `Software\Microsoft\Windows\CurrentVersion\Run`
- 打包：PyInstaller `onedir`
- 测试：标准库 `unittest`

## 架构状态

当前代码已经从单文件 GUI 脚本拆成多个职责较清楚的小模块。`main.py` 仍然保留高风险的窗口生命周期、Win32 热键、窗口激活策略、面板显示/隐藏、数据刷新和应用入口。这是有意保守处理，因为这些逻辑受 Windows 焦点规则影响很大，不能简单按普通 Qt 程序重写。

目前拆出的模块主要是纯逻辑、基础工具或相对独立的 Qt 小组件。这样可以降低 `main.py` 体积，同时尽量不改变关键平台行为。

rounded_widgets.py

  Win32 low-level keyboard hook 使用的 ctypes 结构体和回调类型定义。

win32_constants.py
  Win32 消息、窗口扩展样式、DWM、快捷键和键盘 hook 常量。

## 关键行为边界

下面这些逻辑看起来可能绕，但大概率是为了规避 Windows 焦点、置顶窗口、剪贴板和 Qt 原生窗口之间的边界问题。后续重构时应优先保持行为不变。

- `WS_EX_NOACTIVATE` / `WM_MOUSEACTIVATE`：面板默认不能抢焦点，否则点击符号时可能把目标输入框焦点抢走。
- `SW_SHOWNOACTIVATE` / `SWP_NOACTIVATE`：显示面板时继续避免激活窗口。
- `_visible_panel_hwnd()`：Qt 无边框窗口和实际可见原生窗口句柄之间可能不完全等价，因此用枚举窗口方式定位真实 hwnd。
- `QApplication.processEvents()` 后再应用 Win32 样式：确保 Qt 已创建/显示原生窗口，再改扩展样式。
- `prev_window_handle`：显示面板前保存目标窗口，用于后续粘贴回去。
- `ClipboardUtil.paste_symbol()` 里的短暂 sleep：给剪贴板写入、窗口切回和模拟按键留时间。
- 外部点击检测使用 timer + `GetAsyncKeyState`：普通 Qt focus/blur 不能可靠覆盖 NOACTIVATE 面板。
- 对话框和设置页会临时 `_set_allow_activation(True)`：普通符号面板不抢焦点，但输入框/设置页需要真实键盘焦点。
- 数字键切组使用 low-level keyboard hook：面板 NOACTIVATE 时，普通 Qt 键盘事件未必可靠。

## 当前验证方式

基础静态验证：

```powershell
python -m py_compile main.py settings_view.py symbol_button.py drag_widgets.py rounded_widgets.py resize_handle.py title_bar.py hotkey_edit.py flow_layout.py app_constants.py ui_helpers.py app_logging.py app_paths.py autostart_manager.py hotkey_parser.py data_manager.py config_manager.py clipboard_util.py tests/test_core_logic.py
```

核心逻辑测试：

```powershell
python -m unittest tests.test_core_logic
```

实机人工验证仍然必要，尤其是：

- `Ctrl+Q` 是否能唤起面板。
- 单次模式点击符号后是否关闭面板。
- 连续触发热键是否进入固定模式。
- 点击符号是否粘贴回唤起前的输入框。
- 面板点击是否不抢目标输入框焦点。
- 设置页输入快捷键是否正常。
- 标题栏拖动是否正常。
- 边缘缩放和窗口大小保存是否正常。
- 多屏和负坐标屏幕定位是否正常。
- 符号拖拽排序、跨组移动是否正常。
- 打包后 `data.json` 是否在 exe 同目录可读写。
- 开机自启路径是否符合当前运行方式。

## 当前重构原则

- 小步提交，每次只拆一个明确职责。
- 优先拆纯逻辑、纯 UI 小组件和工具函数。
- 不在同一提交里混入用户数据变更。
- 不随意改 Win32 焦点、热键、剪贴板、粘贴时序。
- 对看似不优雅的 Windows 绕路先加注释，再考虑封装。
- 每轮至少跑 py_compile 和核心 unittest。

## 发布注意

- `data.json` 是默认符号数据，发布包需要保留。
- `config.json` 是用户本机配置，发布包不要预置。
- 不打包 `__pycache__/`、`*.log`、`build/`、`dist/`、`*.spec`。
- 当前采用 PyInstaller `onedir`，输出目录为 `dist/KaomojiPicker/`。
- `build_exe.bat` 会把 `data.json` 复制到 exe 同目录。

## 后续建议

- 先进行一轮实机回归，确认重构后的 UI 组件拆分没有影响显示、点击和拖拽。
- 如果继续重构，下一步可以考虑拆对话框模块，但要保留 `exec_with_activation()` 的焦点策略。
- Win32 热键、窗口激活、粘贴路径应最后拆，并且每次拆分后都要做人工验证。
- 可以逐步扩充测试，优先覆盖 DataManager 的分组重命名、删除、排序，以及 ConfigManager 的异常 JSON 回退。
